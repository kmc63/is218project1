What is SOLID and Design Patterns?

SOLID is an acronym for the five object-oriented design principles. These design principles allow for best use practices that help with developing software to make it easier maintain the project in the long run.

Single Responsibility Principle

In single responsibility principle a class should only have one reason to change or in other words should only have one job. Also, another way to explain it is the class should only have one change that affects the value of the class. If the class is a data base type like being a class of cats or cars it will hold many objects. We would only want it to change the values when we manipulate the data in the objects.

Open-Closed Principle

The open-closed principle in SOLID refers to the idea of the class being open for more functions to be added but closed to modification. To clarify what open and closed means we will not allow modification of an existing class, but we will allow the adding of new functions to an existing class. The main reason we don't want to alter existing code is because we open the door to creating new bugs to what we assume is reliable code. But we can add new functions touching the existing code by using abstract classes.

Liskov Substitution Principle

The liskov substitution principle is the idea that child classes should have no issue being substituted for their parent class. So if class B is a child of class A we should be able to send anu object from class B into a class A method. This should not cause any issues otherwise the liskov substitution principle was not implemented properly.

Interface Segregation Principle

The interface segregation principle is about keeping the interfaces of a class separated. The idea behind the principle is, it becomes more beneficial to have many interfaces to manipulate a class and that having a general purpose interface is bad practice. Another thing about this principle is that a client should never be forced to use functions from a class it doesn't need. An example of this is, a class that has one interface which adds, subtracts, and multiplies with one call. This can be bad because the client may just need to use the add function but is forced to use the subtract and multiply function.

Dependency Inversion Principle

The dependency inversion principle states that our classes should depend upon interfaces or abstract classes instead of static classes and functions. The idea is we want all of our classes to be open to extensions, so instead of our classes depending on static classes we change our classes to become dependent on interfaces.

Design Patterns

Design patterns are the best practices used by experienced object-oriented programmers. This practice can be solutions to a general problem that programmers may come across when developing software. Also, design patterns provide a standard terminology and are specific to particular scenarios. Design patterns are simply best practices when it comes to designing software. These best practices have been created over years from experienced software developers to make software development easier for novice software developers.
